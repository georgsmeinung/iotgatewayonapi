<h1 class="entry-title">IoT Gateway on a Pi</h1>
<div class="entry-content">
<h2 class="wp-block-heading"><strong>TL;DR</strong></h2>



<p class="wp-block-paragraph">As for IoT solutions, communication architecture between transductors and the user interface (UI) is key for having a scalable and flexible platform in place. The almost exclusively used protocolo to communicate edge components with IoT Gateways is MQTT, which brokers are usually hosted on a cloud platform in order to&nbsp; aggregate selected topics with messages from devices and compose dashboards updated in real-time and allows final users to interact with the deployed solutions and most likely have a control rules set in order to automate the solution.&nbsp;</p>



<p class="wp-block-paragraph">This article analyzes how the protocol stack needed for a test IoT solution could be implemented as a single IoT Gateway on a Raspberry Pi.</p>



<h2 class="wp-block-heading"><strong>What is IoT?</strong></h2>



<p class="wp-block-paragraph">The <strong>Internet of Things, or IoT,</strong> refers to the billions of physical devices around the world that are now connected to the internet, collecting and sharing data. Thanks to cheap processors and wireless networks, it&#8217;s possible to turn anything, from a pill to an aeroplane to a self-driving car into part of the IoT. This adds a level of digital intelligence to devices that would be otherwise dumb, enabling them to communicate real-time data without a human being involved, effectively merging the digital and physical worlds.</p>

<h2 class="wp-block-heading"><strong>IoT Architecture</strong></h2>



<p class="wp-block-paragraph">A typical IoT solution is characterized by many devices -thus things- that may use some form of gateway to communicate through a network to a back-end server that is running an IoT platform that helps integrate the IoT information into the existing enterprise. <strong>The roles of the devices, gateways, and cloud platform are well defined</strong>, and each of them provides specific features and functionality required by any robust IoT solution.</p>

<h4 class="wp-block-heading"><strong>Things: Devices (sensors &amp; actuators)</strong></h4>



<p class="wp-block-paragraph">As the basis for every IoT system, the connected <strong>sensors</strong> are responsible for providing the essence of the Internet of Things which is the data. To pick up physical parameters in the outside world or within the object itself, they need sensors. These can be either embedded in the devices themselves or implemented as standalone objects to measure and collect telemetry data. For instance, think of agricultural sensors whose task is to measure parameters such as air and soil temperature and humidity, soil pH levels or crop exposure to sunlight.</p>



<p class="wp-block-paragraph">Another key element of this layer are the <strong>actuators</strong>. Being in close collaboration with the sensors, they can transform the data generated by smart objects into physical action. Let’s imagine a smart watering system with all the necessary sensors in place. Based on the input provided by the sensors, the system analyses the situation in real time and commands the actuators to open selected water valves located in places where soil humidity is below the set value. The valves are kept open until the sensors report that the values are restored to default. All of this happens with or without a single human intervention&#8211; a fully automated action script based upon rules is the most common scenario, though.</p>



<p class="wp-block-paragraph">What is also important is that the connected objects should not only be capable of communicating bidirectionally with their corresponding gateways or data acquisition systems, but also being able to recognise and talk to each other to gather and share information and collaborate in real time to leverage the value of the whole deployment. In case of resource-constrained and battery-operated devices particularly, achieving this is not an easy task since such communication requires lots of computing power and consumes precious energy and bandwidth. Therefore, a robust architecture can only enable effective device management when it uses fit-for-purpose, secure and lightweight communication protocols, such as lightweight M2M which has become a leading standard protocol for the management of low power lightweight devices which are typical for many IoT use cases</p>



<h4 class="wp-block-heading"><strong>The Frontier: Gateways&nbsp;</strong></h4>



<p class="wp-block-paragraph">Although this layer still functions in close proximity with sensors and actuators on given devices, it is essential to describe it as a separate IoT architecture stage as it is crucial for the processes of data collection, filtering and transfer to edge infrastructure and cloud-based platforms.</p>



<p class="wp-block-paragraph">Gateways facilitate communication between the sensors and the rest of the system by converting the <strong>sensor telemetry</strong> <strong>data</strong> into formats that are easily transferable and usable for the other system components behind. Conversely, Gateways also deliver <strong>commands to actuators</strong> originated within the IoT Cloud Control Platform. What’s more, they are able to control, filter and select data to minimise the volume of information that needs to be forwarded to the cloud, which positively affects network transmission costs and response times. Thus, gateways provide a place for the local preprocessing of sensor data which is squeezed into useful bundles ready for further processing.</p>



<p class="wp-block-paragraph">Another aspect that the gateways support is security. Because the gateways are responsible for managing the information flow in both directions, with the help of proper encryption and security tools they can prevent IoT cloud data leaks as well as reduce the risk of malicious outside attacks on IoT devices.</p>



<h4 class="wp-block-heading"><strong>Cloud: The IoT platform</strong></h4>



<p class="wp-block-paragraph">If sensors are neurons and the gateway is the backbone of IoT, then <strong>the cloud is the brain in the Internet of Things body</strong>. A data centre or a cloud-based system is designed to store, process and analyse massive volumes of data for deeper insights using powerful data analytics engines and machine learning mechanisms which edge systems would never be able to support.</p>



<p class="wp-block-paragraph">Having seen increased adoption (especially in Industrial IoT architecture) over the past several years, cloud computing contributes to higher production rates, reduction of unplanned downtime and energy consumption and many other business benefits.</p>



<p class="wp-block-paragraph">If furnished with proper user application solutions, the cloud can provide business intelligence and presentation options that help humans interact with the system, control and monitor it and make informed decisions on the basis of reports, dashboards and data viewed in real time.</p>



<h4 class="wp-block-heading"><strong>Consumers: The Applications</strong></h4>



<p class="wp-block-paragraph">F<strong>inal user can access to the entire IoT Solution through the Applications</strong> -mobile or web- and use them to: have an status of the controlled scenario through the data collected by sensors; ask for an specific action by issuing the proper command; explore controlled scenario insights by using the data generated by the Cloud platform; change, adapt or generated new action rules into the Cloud platform in order to automated the controlled scenario.&nbsp;</p>



<p class="wp-block-paragraph">As for the automated agricultural application mentioned before, the Applications provides the UI to adjust the watering rules, get to know if not foreseen situations are happening and issue not automated commands in order to remediate them.</p>

<h2 class="wp-block-heading"><strong>MQTT Protocol</strong></h2>



<p class="wp-block-paragraph">MQTT is one of the most commonly used protocols in IoT projects. It stands for <strong>Message Queuing Telemetry Transport</strong>. It is designed as a lightweight messaging protocol that uses publish/subscribe operations to exchange data between clients and the server. Furthermore, its small size, low power usage, minimized data packets and ease of implementation make the protocol ideal for the “machine-to-machine” or “Internet of Things” world.</p>



<p class="wp-block-paragraph">MQTT has unique features you can hardly find in other protocols, like:</p>



<ul class="wp-block-list"><li>It’s a lightweight protocol. So, it’s easy to implement in software and fast in data transmission.</li><li>It’s based on a messaging technique. Of course, you know how fast your messenger/WhatsApp message delivery is. Likewise, the MQTT protocol.</li><li>Minimized data packets. Hence, low network usage.</li><li>Low power usage. As a result, it saves the connected device’s battery.</li><li>It’s real time! That&#8217;s specifically what makes it perfect for IoT applications.</li></ul>



<p class="wp-block-paragraph">Like any other internet protocol, MQTT is based on clients and a server. Likewise, the server is the component responsible for handling the client’s requests of receiving or sending data between each other. MQTT server is called a broker and the clients are simply the connected devices, so:</p>



<ul class="wp-block-list"><li>When a device (a client) wants to send data to the broker, we call this operation a “publish”.</li><li>When a device (a client) wants to receive data from the broker, we call this operation a “subscribe”.</li></ul>



<p class="wp-block-paragraph"><img loading="lazy" src="https://lh3.googleusercontent.com/zz5bfN5yxGUt6j7_yYPZKeKgfizA85IX1Kjdtzw7AunkxZ5OYgU3vdkM6flPVoQK4ue79rh5HB-8sms3z6GLoXC8bLEhsAYO8Q7G9DlETz2qzRoD-0I1nlysC8A3Qym57BWvs0g" width="436" height="224"></p>



<p class="wp-block-paragraph">As shown, the MQTT components are 5 as follows:</p>



<ol class="wp-block-list"><li><strong>Broker</strong>, which is the server that handles the data transmission between the clients.</li><li>A <strong>topic</strong>, which is the “logical place” a device wants to put or retrieve a message to/from.</li><li>The <strong>message</strong>, which is the data that a device receives “when subscribing” from a topic or sends “when publishing” to a topic.</li><li><strong>Publisher</strong> is the process a device does to send its message to the broker.</li><li><strong>Subscriber</strong> is a device that does receive a message from the broker.</li></ol>



<p class="wp-block-paragraph">Conversely to MQTT, HTTP is slower, more overhead and power consuming protocol than MQTT. So, let’s get into each one separately:</p>



<ul class="wp-block-list"><li><strong>Slower</strong>: because it uses bigger data packets to communicate with the server.</li><li><strong>Overhead</strong>: HTTP request opens and closes the connection at each request, while MQTT stays online to make the channel always open between the broker “server” and clients.</li><li><strong>Power consuming</strong>: since it takes a longer time and more data packets, therefore it uses much power</li></ul>

<h2 class="wp-block-heading"><strong>General Solution Description</strong></h2>



<p class="wp-block-paragraph">Functionally the proposed solution emulates a domotic control central intended to manage home devices. In order to connect to the solution, IoT nodes use a microcontroller board with a program that implements a MQTT client.</p>



<p class="wp-block-paragraph">As for the actuator part of the IoT device, it’s reading to regular intervals its status topic from the MQTT Broker and reacts turning the controlled equipment on or off. In the prototype this actuation action will be emulated by turning on or off the red LED. In a real control scenario the actuator parte will open or close a relay by using an optocoupler module.</p>



<p class="wp-block-paragraph">On the other hand the sensor part of the IoT module will write to regular intervals the temperature and humidity readings into a topic in the MQTT broker.</p>



<p class="wp-block-paragraph">This kind of home device could be powered by using 220V AC to 5V CC power adapters.</p>



<p class="wp-block-paragraph">The central in this domotic deployment is implemented with a Raspberry Pi Zero W -thus, the IoT Gateway- by using a MQTT Broker -Mosquitto MQTT-, a Rule Engine -Node-RED- and a backend MQTT client -with a Node-RED module. The MQTT broker service implements the topics of sensor reading and actuator status. The Rule Engine service implements a watcher of the controlled temperature and humidity topics and changes the state of the actuator topic if the programmed threshold is reached. Also as part of the service provided by the Node-RED server a web user interface allows both to graphically configure the Rule Engine and to implement the User Interface Control Dashboard which turn to be of responsive design.In a typical domotic control scenario, an interaction between the sensor and the actuator is configured in order to activate a humidifier if the humidity reading drops below a given threshold. This is implemented in a Node-RED flow connecting a <strong>“network/mqtt in” </strong>node, a <strong>“function/function” </strong>node -which internally implements the condition evaluation- and a <strong>“network/mqtt out”</strong> as follows:</p>

The humidity value read is receive in the <strong>msg.payload</strong> message property, changed accordingly to the condition and pass forward to the&nbsp; <strong>“network/mqtt out”</strong> node as shown below:</p>

<h2 class="wp-block-heading"><strong>Detailed communication description</strong></h2>



<p class="wp-block-paragraph">In this section the complete global connection and deployment of the solution is explained to a low level all the way from the sensor to the integration into the dashboard. A similar circuit must be implemented for each actuator to be used into the solution adapting in each case:</p>



<ul class="wp-block-list"><li><strong>Connection to the sensor or the actuator with the controller board. </strong>Each transductor has a particular connection mode and depending on its datasheet specification it should be adapted to the connection with the controlling pins of the controller device.</li><li><strong>Controller board routine for the spaced sensor reading of writing for the activating actuators. </strong>Although the solution presented tried to simplify the connection used in order to read or write a digital transductor channel, sometimes it could be necessary to implement a more complex access routine to analog transductors by using, for instance, PWM -Pulse-width modulation- or another specific protocols such us I2C -Inter-Integrated Circuit- or SPI -Serial Peripheral Interface-.&nbsp;</li><li><strong>Controller board routine for suitable media access protocol. </strong>In the case of the NodeMCU board used in the solution, it provides the WiFi modules required for wireless link to the IoT Gateway so WiFi protocol libraries are required; but with other boards -such us Arduino- it requires using an extra Ethernet Shield for wired connection, and so it requires the suitable Ethernet protocol library.</li><li><strong>Controller board routine that implements the required MQTT client. </strong>In the case of this solution the proper MQTT client library in order to interact with the Mosquitto MQTT Broker is the one that suits the NodeMCU WiFi module. If another IoT cloud platform is used, this routine should be changed accordingly in order to make the IoT device speak the required protocol when interacting with platforms like Cayenne or Amazon IoT.</li><li><strong>MQTT Broker for the solution</strong>. In this solution the MQTT Broker is implemented with open source Mosquitto Broker, but another equivalent piece of software could be used instead with a local IoT Gateway or its cloud equivalent such as Amazon AWS IoT, Google Cloud IoT, Azure IoT Hub, etc.</li><li><strong>Dashboard rendering for the solution. </strong>In this solution it’s used Node-RED for implementing the topic reading, control and visualization User interface. If another IoT Cloud Platform is used instead, it generally implements all this functionality and possibly more in its own fashion.</li></ul>

<h4 class="wp-block-heading"><strong>Mosquitto MQTT Broker on the Pi Zero W</strong></h4>



<p class="wp-block-paragraph">On theRaspberry Pi server Mosquitto MQTT service must be installed and its starting automated in OS startup routines according to Mosquitto and Raspbian documentation.&nbsp;In order to test the proper function of Mosquitto, we start a SSH connection and use Mosquittto CLI command in order to subscribe and listen to the test topic “hello_topic” as follows:</p>


<div class="wp-block-syntaxhighlighter-code "><pre class="brush: bash; gutter: false; title: ; notranslate" title="">
mosquitto_sub -d -t hello_topic
</pre></div>


<p class="wp-block-paragraph">In a separate SSH connection, similarly to what NodeMCU program does, we can publish a message into the test topic “hello_topic” with the following Mosquitto CLI command:</p>


<div class="wp-block-syntaxhighlighter-code "><pre class="brush: bash; gutter: false; title: ; notranslate" title="">
mosquitto_pub –d –t hello_topic –m "hello from RaspberryPi"
</pre></div>


<p class="wp-block-paragraph">In the first SSH connection we should visualize the message “hello&nbsp;from&nbsp;RaspberryPi” written into the test topic “hello_topic”, since we’re watching its live activity there.</p>



<h4 class="wp-block-heading"><strong>Node-RED implementation on Raspberry Pi</strong></h4>



<p class="wp-block-paragraph">In this solution we used Node-RED in order to implement a flow for reading and publishing control topics and visualize topic values read and written. Node-RED is a flow engine -written for NodeJS, thus the name- with a IoT approach which allows to graphically design flow services accessible by a RESTful API, MQTT, Websocker &amp; AMQT among other protocols. Besides Node-RED provided integration with 3rd-party APIS such as Twitter, Facebook and Yahoo!. Along with the flow engine, Node-RED provides a visual tool that can run on a Raspberry but also in other cloud platforms such as AWS IoT or Azure IoT.&nbsp;&nbsp;&nbsp;</p>



<p class="wp-block-paragraph">Among its many plugins Node-RED provides an extension that allows to create and render dashboards with IoT Devices signals received through several protocolos adapters. This Node-RED dashboard plugin will be used as the UI of the proposed IoT solution.</p>



<p class="wp-block-paragraph">So in order to visualize the value of a given MQTT topic a dashboard containing connectors to the topics “<strong>/casa/temperatura</strong>” -/house/temperature/- and&nbsp; “<strong>/casa/humedad</strong>” -/house/humidity- and even we can show the actuator status value in the topic “<strong>/casa/humidificador</strong>” -/house/humidifier- all of then publish in our local MQTT Broker with the following configuration:</p>

<h2 class="wp-block-heading"><strong>Conclusions and further improvement</strong></h2>



<p class="wp-block-paragraph">The presented IoT solution is intended to be a Proof of Concept for a complete installation of all the components involved in an IoT implementation&#8211;from the sensor to the final user interface visualization.</p>



<p class="wp-block-paragraph">It wasn’t used any ready-to-use IoT cloud platform purposely in order to check all the layers involved and to to delegate any of the technology stack responsibilities even thes case of the MQTT broker and the dashboard rendering.</p>



<p class="wp-block-paragraph">For a commercial solution there should be more redundant components in order to improve the overall availability&#8211;thus using a cloud MQTT Broker, data collection repository and dashboard rendering engine is advisable. Still if not using a cloud platform, some additional security measures must be implemented on the Raspberry Pi Zero W in order to connect it to the Internet.</p>



<p class="wp-block-paragraph">The presented IoT device was built upon a hobbyist breadboard plus a NodeMCU Dev Kit; a final version should be designed as a specific board optimized and battery powered; a more optimized version should have a much smaller footprint too. Having the final IoT device powered by battery introduces a new variable -the battery level- that should be considered in order to planned preventive maintenance of the deployed solution.</p>



<p class="wp-block-paragraph">Finally the presented scenario could gain further insight&nbsp; if a history module is added in order to generate historic data profiles and possibly explode them with Data Science tools.</p>
